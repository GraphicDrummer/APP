## 1일차 (07-03) — 환경 구축과 저장 계층

### 1-1. 환경 구축에서 만난 것들

저장소 클론 → `npm install` 시도 → **`npm: command not found`**.
Node.js 자체가 없었다. 해결 과정:

1. `winget install OpenJS.NodeJS.LTS` (v24.18.0)
2. 설치 직후엔 기존 터미널 프로세스가 PATH를 모름 → 명령마다 PATH를 명시하거나 새 터미널
3. 개발 서버(preview 도구)가 `'node'은(는) 내부 또는 외부 명령...` 에러 —
   npm.cmd가 내부에서 PATH의 node를 찾기 때문. **launch.json에서 node.exe 절대경로로
   vite를 직접 실행**하는 방식으로 해결
4. 이후에도 고아 프로세스가 5173 포트를 점유하는 일이 두어 번 —
   `Get-NetTCPConnection -LocalPort 5173`으로 PID 찾아 종료

교훈: "설치했는데 안 된다"의 대부분은 PATH 문제이고, 이미 떠 있는 프로세스는
새 설치를 모른다.

### 1-2. Supabase 저장 계층 (`b3bcd77`)

- 테이블 3개: `meetings`(모임) / `participants`(참여자) / `availability`(가용 시간)
- 별도 백엔드 서버 없이 브라우저에서 supabase-js로 직접 통신하는 구조 선택
- `.env`로 키 분리 (`.gitignore`에 추가), `.env.example`만 커밋
- RLS는 "링크 기반 서비스라 아직 인증이 없다"는 이유로 anon 전면 허용으로 시작
  — **이 결정이 6일차에 보안 사고로 돌아온다** (7장 참고)

**첫 번째 기술 함정**: supabase-js의 `Database` 제네릭 타입을 `interface`로
정의하면 모든 쿼리 결과가 `never`가 된다 (TS2353 에러 다발). interface에는
암시적 인덱스 시그니처가 없기 때문. **`type` 별칭으로 바꿔야 동작.**
`database.types.ts` 상단에 주석으로 박제해뒀다.

**개발 프로세스 사건 2건**:
- main 직접 push가 Claude Code의 권한 분류기에 거부됨 → 사용자가 명시적으로
  허용 의사를 밝힌 뒤 통과. 이후 매번 묻지 않도록 `.claude/settings.json`에
  허용 규칙 추가를 시도했으나 **AI가 자기 권한을 스스로 넓히는 것은 거부됨**(3회 시도
  모두) → 사용자가 메모장으로 직접 파일 작성. "권한은 사람이 부여한다"는 설계.
- git 사용자 정보 미설정(`Author identity unknown`) → 저장소 로컬로
  `GraphicDrummer <doyomee@gmail.com>` 설정.

**협업 패턴 확립**: DDL(테이블 생성 등)은 anon 키로 실행 불가 →
"AI가 마이그레이션 SQL 파일 작성 → 사람이 Supabase 대시보드 SQL Editor에서 실행"
분업이 이때 만들어져 끝까지 유지됨.

## 4일차 오전 (07-05~06) — 관리자 링크와 두 번의 위기

### 4-1. 세션 중단과 진단 보고서

관리자 링크 작업 도중 **사용량 한도로 세션이 끊김**. 그 사이 사용자는 Vercel
배포를 별도로 진행. 새 세션에서 사용자가 요청한 것: "고치지 말고 **진단 보고서만**."

진단으로 드러난 것 (git 상태/빌드/코드/배포 4개 축):

1. 관리자 링크 작업이 **미커밋 상태로 절반만** 존재. 그런데 접근 방식이 도중에
   바뀌어 있었음: 원래 설계(`/admin/:code` 경로 + `admin_code` 컬럼)와
   실제 코드(`/m/:code?adminKey=` 쿼리 + `admin_key` 컬럼)가 혼재
2. **build 스크립트에서 `tsc -b`가 제거**되어 타입 에러가 은폐되고 있었음
3. `db.ts` 전반이 `Promise<any>`, `as any` 캐스팅으로 도배 — 타입 안전망 무력화
4. 라이브 DB에는 `admin_key` 컬럼이 이미 수동 생성돼 있고(마이그레이션 파일 없이),
   문서(schema.sql/types/마이그레이션)는 전부 `admin_code`로 어긋남 (스키마 드리프트)
5. `App.tsx`에 `/m/:code` 라우트 중복, 죽은 함수 `getMeetingByAdminCode`
6. **보안 구멍 실증**: curl로 anon 키만 가지고
   `select title, admin_key from meetings` → **8개 모임의 관리자 키가 통째로 응답됨**
7. Vercel: `vercel.json` 없음(SPA 딥링크 404 위험), 환경변수 미설정

교훈: 중단된 작업을 이어받을 때는 "어디까지 됐지?"가 아니라
**"지금 상태가 스스로 모순되지 않는가?"** 를 먼저 검사해야 한다.

### 4-2. 보안 수정 (`b425466`) — 이 프로젝트 최대의 기술 사건

문제의 본질: 프론트엔드에 박히는 anon 키는 **원래 공개돼도 되는 값**이다.
따라서 "anon으로 뭘 할 수 있는가"가 곧 "전세계 누구나 뭘 할 수 있는가"다.
`using (true)` RLS는 그 전제에서 admin_key 열람 + 임의 행 UPDATE를 모두 허용하고 있었다.

수정 (마이그레이션 0006):

1. anon의 meetings SELECT 권한 회수 → **admin_key를 제외한 컬럼 목록만** 재부여
   (Postgres 컬럼 단위 GRANT)
2. anon의 meetings UPDATE 완전 회수
3. `SECURITY DEFINER` 함수 3개 신설 (함수 안에서만 소유자 권한으로 실행):
   - `verify_admin_key(share_code, admin_key)` — 둘 다 맞아야 행 반환
   - `admin_update_meeting_info(...)` / `admin_set_confirmed_slot(...)` —
     admin_key가 맞는 행만 갱신
4. 프론트의 `isAdmin` 판정을 클라이언트 문자열 비교가 아니라
   **"서버 검증을 통과해 admin_key가 채워진 응답을 받았는가"** 로 변경
5. db.ts에 `PUBLIC_MEETING_COLUMNS` 명시 목록 — select()가 실수로 전체 컬럼을
   요구하지 않도록

실행 중 사건: `ERROR: operator does not exist: uuid = text` —
라이브 컬럼이 uuid 타입이라 text 파라미터와 직접 비교 불가 →
`admin_key::text = p_admin_key` 캐스팅으로 해결.

검증 (수정 전후 모두 curl 실측):
- `select admin_key` / `select *` → `42501 permission denied` ✅
- 직접 PATCH → `42501` ✅
- `verify_admin_key` 틀린 값 → `[]`, 맞는 값 → 정상 반환 ✅

동시에 진단에서 나온 부채 전부 청산: build 스크립트 복원, any 전면 제거,
admin_code→admin_key 문서 통일, 중복 라우트/죽은 코드 삭제.
`slot.slot_datetime ?? slot.h` 같은 무의미 코드(항상 폴백돼 우연히 동작)도 정리.

확립된 원칙: **"에러를 숨기지 않는다. any와 빌드 우회는 반드시 더 크게 돌아온다."**

### 4-3. 입력 모델 반전 (`1151f56`) — 제품 철학의 전환

  recommend/사다리/병목은 집계값 의미가 보존되므로 무변경
- **프리셋 재작성의 묘수**: "전원을 모든 슬롯 가능으로 채운 뒤, 안 되는 시간만
  지우는" 방식으로 재구성 → 슬롯별 가용성이 이전과 바이트 단위로 동일 →
  **기존 엔진 테스트 7개가 수정 0줄로 통과** = 동치 변환의 증명

  (사용자 사전 승인) — 마이그레이션 0007
- 저장 규약 유지: "표시된 것만 저장하는 희소 구조" — 무엇이 암묵값인지만 반전

동시 수정 — **칩 스크롤 버그**: 사용자 제보 "드래그 스크롤이 안 되고 클릭하면

있었다. **마운트 1회만 실행**으로 변경 + 마우스 드래그 스크롤 추가
(드래그 종료 시 밑의 칩이 클릭되지 않게 moved 플래그 처리).

말투 작업: "🐭 고민 중", "모두가 OO님만 기다리고 있어요!",

### 5-2. "배포했는데 옛날 코드" 미스터리 — 원인이 4겹

사용자가 Vercel 배포를 직접 진행했는데 수정 전 동작(칩 스크롤 안 됨)이 보인다고 제보.
스크린샷을 하나씩 받아 진단한 결과:

**겹 1 — 환경변수 없음**: Environment Variables가 완전히 비어 있었음.
`supabase.ts`는 값이 없으면 throw하므로 사이트 자체가 깨질 상태.
입력 과정에서도 사건: **Key 칸에 URL 값을 입력** → "invalid characters" 에러.
Key=변수이름, Value=값. "Import .env" 기능 안내로 해결.
Sensitive 토글 시 Development 환경 잠김은 정상 동작(로컬은 .env 파일 사용)임을 설명.

**겹 2 — GitHub 기본 브랜치가 엉뚱함**: 초기 셋업 부산물인
`claude/recommendation-engine-setup-gknema`(커밋 2개짜리)가 Default branch로
남아 있었고, main은 18커밋 앞서 있었음 → Settings → General에서 main으로 변경.

**겹 3 — 진짜 원인: Vercel Drop**: 배포 목록을 뜯어보니 최초 배포에
"Vercel Drop"(수동 폴더 업로드) 표기, 이후 전부 "Redeploy of ..." 체인.
**Git 기반 배포가 한 번도 없었다.** 결정적 증거: 배포 Source에 `dist/`와
`.env`가 보임 — .gitignore된 파일이라 Git 배포에선 절대 나올 수 없음.
빌드 시간 7초(정적 업로드)도 방증. Redeploy는 "그 배포 시점의 스냅샷"을
재빌드하므로 아무리 눌러도 코드가 안 바뀌었던 것.

**겹 4 — git 명령을 엉뚱한 폴더에서**: 안내한 빈 커밋 push를 사용자가
`C:\Users\KWONDO`(홈)에서 실행 → `fatal: not a git repository`.
저장소는 `D:\!CLAUDE\APP`.

**해결**: 기본 브랜치 변경(사용자) + Git 재연결(사용자) + 저장소 폴더에서
`git commit --allow-empty -m "chore: trigger fresh deploy from main"` push(AI)
→ **첫 진짜 Git 배포 성공**. 이후 push만 하면 자동 배포되는 정상 파이프라인 확립.

**부수 지식**: Deploy Hooks = "이 URL을 호출하면 지정 브랜치를 배포해줘"라는
비밀 링크. CMS 연동/정기 재빌드/비상 수동 트리거용. 이번엔 불필요.

**정리 권장사항**: 옛 Drop 배포들에는 .env가 포함돼 있음(anon 키라 실해는
없지만) → 새 Git 배포가 Production이 된 뒤 삭제 권장.
